# AGENTS.md

## 项目简介

theNanGuos 是一个基于 LangGraph 的本地多智能体音乐创作实验项目。系统将多个音乐角色组织成“乐团”，由 Conductor Agent 理解用户需求并选择工作流，其他 Agent 分别完成作词、作曲、编曲等任务，最后生成音乐模型提示词、可选的乐谱或 demo 音频，并调用 Suno 兼容服务生成音乐。


## 技术方向

- Python 3.12，使用 `uv` 管理依赖和运行命令。
- 使用 LangGraph 编排 Agent 和条件分支。
- 使用 Pydantic 定义 Agent 的结构化输入与输出。
- 后端计划使用 FastAPI，前端计划使用 React、TypeScript 和 React Flow。
- 项目数据、工作流、上传文件和生成产物均保存在本地文件系统。
- MusicXML、MIDI 和 demo 音频是可选产物，不要求每种音乐工作流都生成。

## 实现约束

- 优先保持实现简单、可读、可本地运行，不引入数据库、消息队列、容器编排或生产级基础设施。
- Agent 之间通过明确的结构化状态传递领域数据，不依赖解析自由文本消息获取关键字段。
- 每个 Agent 只读取完成任务所需的状态，并只更新自己负责的字段。
- Conductor 只从系统允许的工作流和角色中进行选择，不允许模型生成或执行任意代码。
- 不同音乐类型使用可组合的工作流；纯器乐工作流必须能够跳过 Lyrics Agent。
- 将音乐领域职责与供应商适配分离。例如编曲由 Arranger Agent 负责，Suno 提示词由 Prompt Compiler 负责。
- Suno 等外部服务通过独立 provider 封装，避免将供应商请求逻辑写入 Agent 或图节点。
- 用户自定义工作流使用受约束的 JSON 定义，只允许引用系统已注册的节点类型。
- API 密钥和服务地址只从环境变量读取，不提交密钥、生成音频或本地运行数据。
- 修改依赖时同步更新 `pyproject.toml` 和 `uv.lock`。
- 保持改动聚焦；除非当前功能需要，不做无关重构。

## 本地文件约定

- `agents/`：Agent 实现。
- `app/`：LangGraph 图和应用编排。
- `models/`：状态及结构化领域模型。
- `prompts/`：Agent 系统提示词。
- `lib/`：外部服务和音乐工具封装。
- `data/`：本地项目、运行状态和用户资产，默认不提交。
- `works/`：生成的音乐产物，默认不提交新文件。

## 代码质量

- 新增公共行为应提供聚焦的自动化测试。
- 外部 LLM、Suno 和音频工具在测试中使用 mock 或 fake，测试不得产生真实付费请求。
- 优先使用异步接口处理网络请求，但不要为了形式统一而增加无必要的抽象。
- 错误信息应包含失败阶段和可操作原因，避免静默忽略结构化输出校验失败。
- 完成改动后至少运行相关测试；没有测试时运行：

```bash
uv run python -m compileall -q agents app lib models main.py
```
